# Comedy Show Host Tool — Requirements Specification

**Version:** 1.0 (Draft)
**Date:** 2026-07-15
**Audience:** Software developer working with Claude Code. Written to be both human- and LLM-readable.

---

## 1. Vision

A web-based toolkit for running live comedy lineup shows and open mics. It keeps every performer on time, gives the host full control of the evening, and (in later phases) captures a timestamped record of each act — transcription, laughter, and crowd-work insights — that performers waiting backstage can use in real time.

**Guiding principles:**

- **Web only.** No native apps, no App Store. All clients are web pages joined via URL / QR code.
- **One server, many ephemeral clients.** The server holds show state; clients connect and disconnect freely. The server never tracks devices — only open web sessions.
- **No audio over the network.** All speech-to-text and laughter detection runs locally on the capturing device. Only text + metadata (timestamps, confidence, laughter intensity) is sent to the server.
- **Free / open-source AI only.** Whisper, SenseVoice, or similar local models. No paid transcription APIs. No LLM-as-a-service in core flows (rule engine first; LLM enrichment is an optional future add-on).
- **Consent-aware.** Performers can opt out of having their set transcribed/recorded.

## 3. System Overview

```
                        +--------------------+
                        |       SERVER       |
                        | show state, clock  |
                        | merge engine (P2)  |
                        | insights (P1+)     |
                        +---------+----------+
              real-time sync (WebSocket) |
     +--------------+-----------+-----------------+---------------+
     |              |           |                 |               |
+----+-----+  +-----+------+ +--+-----------+ +---+------------+
| HOST APP |  | STAGE      | | LAUGH TRACKER| | FOLLOW VIEW    |
| (phone)  |  | DISPLAY    | | (back of     | | (performers'   |
| P0       |  | (iPad/PC)  | |  room) P2    | |  phones) P1    |
|          |  | P0 timer   | |              | |  read-only     |
| 3 tabs   |  | P1 mic+STT | | mic + laugh  | |  lineup +      |
|          |  |            | | detection    | |  insights      |
+----------+  +------------+ +--------------+ +----------------+
```

- The **host client is mandatory** — without it there is no show. All other clients are optional consumers.
- All timers everywhere derive from the **single master clock controlled by the host** (start / pause / next).
- Clients join via a **QR code / share link** generated by the host (no accounts in MVP).

## 4. Clients & Features

### 4.1 Host App (mobile browser) — P0

The control center. Three tabs, always visible:

#### Tab 1: Plan (P0)

Build and edit the show skeleton.

- Add performers: name + set length in minutes. *(P0)*
- Add breaks with a duration. *(P0)*
- Per-act warning threshold ("red light"), default 1 minute before end, customizable per act or show-wide. *(P0)*
- Drag-and-drop reordering of acts. *(P0)*
- Delete an act with one tap (X next to name) — open-mic cancellations are common. *(P0)*
- **No separate save step.** The skeleton is live the moment it's entered; the host can build it two minutes before the show and hit Start immediately. *(P0)*
- **Live editing during the show.** Editing the plan (reorder, delete, adjust times of *upcoming* acts) never affects the currently running act's timer or its transcription track. *(P0)*
- Per-performer consent toggle: "record/transcribe this set" on/off. *(P1)*

#### Tab 2: Show (P0)

Run the show.

- **Start** — begins the show / the next act. *(P0)*
- **Next** — single tap cuts over to the next act in the (possibly just-edited) lineup. *(P0)*
- **Pause / Resume** — pauses the master clock; every connected client's timer pauses in sync. *(P0)*
- Display: current act, elapsed / remaining time, who is up next. *(P0)*
- Between acts, the show is implicitly in a "host segment" — the host is on stage doing bits. This segment is its own track (timed; transcribed in P2 when host-device transcription lands). *(P0 for timing, P2 for transcription)*
- Auto-run option: if the host does nothing, acts advance automatically when their time (incl. any configured buffer) elapses. *(P1)*
- Rolling view of incoming transcription / laughter data for the current act (slight delay is fine — this is not a live teleprompter). *(P1)*

#### Tab 3: Follow (P1)

Identical to the performers' Follow view (see 4.4), embedded as a read-only tab so the host sees what performers see.

### 4.2 Stage Display (iPad or PC, on stage facing the performer) — P0

A giant, unmissable timer. **The performer never touches it.**

Visual states *(P0)*:

| State | Visual |
|-------|--------|
| Normal running time | **White clock digits on black background** |
| Final minute (warning threshold reached) | Screen turns **red**, clock digits **black** |
| Overtime (0 to −1:00) | Screen **blinks red ↔ black** |
| Overtime beyond 1 minute | Escalating multicolor blinking: white/black, yellow/black, red/black cycling — impossible to ignore |

- Clock digits as large as the viewport allows. *(P0)*
- Follows the master clock exactly: host's pause pauses it, Next resets it for the new act. *(P0)*
- Optional small line: current performer name / next up. *(P1)*
- **Microphone capture + local transcription and laughter detection** of the performer, sending text + timestamps + laughter events (never audio) to the server. Runs only when the current act has consented. *(P1)*
- A **fresh transcription track starts on every host Start/Next** — one clean, separate track per performer (and per host segment), not one huge show-length file. *(P1)*

### 4.3 Laugh Tracker (device at the back of the room) — P2

- Dedicated web client on a device with good audio pickup positioned in the audience.
- Runs local laughter detection (intensity + timestamps); optionally local transcription of audible crowd interactions. Sends only data, never audio. *(P2)*
- Feeds a laughter-intensity timeline per act to the server. *(P2)*

### 4.4 Follow View (performers' phones, read-only) — P1

For comedians waiting in the back room who can't hear the stage.

- Mini read-only skeleton: lineup, who's on now, who's next, timing. *(P1)*
- **Insights — MVP scope: locations only.** A simple list of location names mentioned during the show so far. *(P1)*
  - Deliberately minimal: it does **not** say who mentioned the location or in what context. This is a feature, not a gap — it forces the comedian to check with the host ("was someone here from Boston?" — "no, performer one was talking about visiting Boston"). Keeps backend logic simple and foolproof.
- Later insights *(P2)*: distinguish **audience-mentioned** vs **performer-mentioned** entities; add professions and other crowd-work hooks; per-act topic summaries so the next act knows what material is burned.

## 5. Server

### 5.1 Show state & real-time sync — P0

- Holds the single source of truth: skeleton (ordered acts with durations + warning thresholds), master clock state (running / paused / current act / elapsed), and show history.
- Pushes state changes to all connected clients in real time via **WebSocket** (server-push; clients never poll).
- Stateless about devices: any number of clients of any type may be connected or not.
- Join model: host creates a show → gets a share URL + QR code → other clients open role-specific URLs (e.g. `/show/{id}/stage`, `/show/{id}/follow`). *(P0)*

### 5.2 Track storage — P1

- One transcription + laughter track per act (and per host segment), keyed to the show and act, with wall-clock and act-relative timestamps.
- Exportable per performer (so sets can be shared back to comedians later). *(P2)*
- Respect consent: no track is stored for opted-out performers. *(P1)*

### 5.3 Merge engine — P2

Multiple devices (stage display, host phone, laugh tracker) may transcribe the **same audio** and send **competing interpretations**.

- Server reconciles 2–3 parallel text streams for the same time window.
- Strategy: weighted voting — where two devices agree on a phrase, that wins; otherwise prefer highest confidence; keep alternatives flagged for later review. *(P2)*
- Laughter: blend intensity scores across sources or take the highest-confidence signal. *(P2)*

### 5.4 Insights extraction — P1 (rule engine), P3 (LLM)

- **P1:** simple rule engine — match transcribed tokens against a gazetteer of real locations (cities, countries). Output: deduplicated list of locations mentioned, pushed to Follow views.
- **P2:** extend rule engine to professions and crowd-work patterns ("where are you from?", "what do you do?"); tag whether the entity came from an audience interaction or the performer's material.
- **P3 (optional, off by default):** LLM (e.g. Claude API) for context-aware correction ("Bos-on" → "Boston"), fuzzy classification ("I work in tech"), and act summaries. Explicitly excluded from core flows to avoid running costs.

## 6. Transcription & Laughter Tech (local, free) — P1/P2

Requirements for the on-device models:

- **Speech-to-text:** OpenAI **Whisper** (open source) or **SenseVoice** (open source; also detects laughter/applause/emotion in one pass, and is faster than Whisper). Candidates for browser execution: whisper.cpp compiled to WASM, transformers.js, or the browser Web Speech API as a low-effort P1 fallback (note: Web Speech API may route audio to a vendor service — evaluate against the "no audio leaves the device we don't control" principle before using it beyond prototyping).
- **Laughter detection:** SenseVoice's built-in events, or a dedicated model such as **LaughterSegmentation** (open source, 2024) for laughter onset/offset timestamps.
- Output format per device, per act: `{actId, deviceRole, segments: [{tStart, tEnd, text, confidence}], laughs: [{tStart, tEnd, intensity, confidence}]}`.
- Latency target: near-real-time with a few seconds' delay acceptable (the Follow/host views are not teleprompters).

**Developer note:** running Whisper-class models inside a mobile browser is the riskiest technical assumption in this spec. Prototype this early (P1 spike). Acceptable fallback: a small local companion process/native shell on the stage device only — but the architecture contract (text-only to server) must hold regardless.

## 7. Non-Functional Requirements

- **Simplicity of setup:** a host with zero training can build a 10-act skeleton and start the show in under 3 minutes. *(P0)*
- **Resilience:** if any client disconnects (backstage Wi-Fi…), it re-syncs full state on reconnect; the show clock lives on the server and never depends on a client staying open. *(P0)*
- **Readability:** stage display legible from ~5 m under stage lighting. *(P0)*
- **Privacy:** no raw audio transmitted or stored, ever; opted-out acts produce no tracks. *(P1)*
- **Cost:** zero per-show running cost for AI features (local models only). *(P1)*
- Rough scale: one show at a time per server instance is fine for v1; tens of connected clients max.

## 8. Suggested Build Order (for Claude Code)

1. **P0.1** — Server: show model, master clock, WebSocket state broadcast, show/join URLs + QR.
2. **P0.2** — Host app: Plan tab (add/reorder/delete acts, breaks, thresholds, live editing) + Show tab (start/next/pause, current/next act display).
3. **P0.3** — Stage display: full-screen timer with the four visual states, synced to master clock.
   → **Demo-able MVP: run a real open mic on timing alone.**
4. **P1.1** — Spike: in-browser Whisper/SenseVoice feasibility on iPad.
5. **P1.2** — Stage device transcription + laughter, per-act tracks, consent toggle.
6. **P1.3** — Rule-engine location extraction; Follow view (read-only lineup + locations list); Follow tab in host app; rolling transcript in host Show tab; auto-run mode.
7. **P2** — Laugh tracker client, host-device transcription of host segments, merge engine, richer insights (professions, audience-vs-performer attribution), per-performer export.
8. **P3** — Optional LLM enrichment layer (opt-in, cost-gated).

## 9. Open Questions

- Exact blink timings/colors for overtime escalation (current spec: designer's discretion within the described pattern).
- Whether breaks show anything on the stage display (blank? "Break — back at 21:15"?).
- How long tracks are retained on the server and who can download them.
- Auth: is an unguessable show URL enough, or does the host link need a PIN? (Recommend: host URL secret, viewer URLs harmless.)
  - **Decision (P0.1):** host URL secret token, viewer URLs harmless.

## Appendix: P0.1 implementation decisions (2026-07-15)

Recorded here because the spec left them open:

- **Start/Next semantics.** `Start` begins the next act (from idle or a host
  segment). `Next` during an act ends it and enters a *timed host segment*;
  `Next` during a host segment brings on the next act. So a host who only
  ever taps Next gets act → host → act → host…, and the "single tap cutover"
  of the spec is two quick taps through a momentary host segment. Revisit in
  P0.2 if it feels wrong on a real night.
- **Running act snapshot.** When an act starts, its name/duration/threshold
  are copied into the clock segment, which is how live lineup edits are
  guaranteed never to affect the running timer.
- **Multiple concurrent shows** are supported (trivial with the room model),
  even though v1 only needs one.
